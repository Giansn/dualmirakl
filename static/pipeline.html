v4 problem: sequential phases

Each tick runs A→B→C→F→D in order.
GPU 0 waits while GPU 1 responds.
GPU 1 waits while CPU scores.
GPU 2 runs a 1ms FLAME burst then sleeps.
Result: ~55% peak, ~35% average utilization.

the biggest gap: batch-then-score

v4 collects all N responses, then scores them all at once. CPU sits idle during the entire response phase. Both GPUs sit idle during the entire scoring phase. This is where ~40% of the cycle is wasted.

FLAME: burst vs continuous

v4 runs FLAME once per tick — a single step() call that takes <1ms on GPU 2. The GPU is allocated but doing nothing for 99% of the time. It could be running a continuous population loop instead.

stream-and-score (opportunity 1)

Each agent's response is embedded and scored the moment it arrives via asyncio.as_completed. CPU works in parallel with GPUs. No more batch-then-wait.
GPU 1: 50%→80% · CPU: 20%→55%

speculative stimuli (opportunity 2)

While GPU 1 generates responses, GPU 0 pre-generates next tick's stimuli using the current scores. If scores don't change much (common), the cache hits and tick N+1 starts instantly. If they diverge, re-generate with Ahit (fast delta).
GPU 0: 55%→75%

continuous FLAME (opportunity 3)

GPU 2 runs a perpetual while True: step() loop. LLM scores are injected into influencer agents asynchronously whenever they arrive (vertical lines). Population evolves thousands of steps between ticks instead of one.
GPU 2: 1%→90%

v4 → v5: what changes

gather() → as_completed()
Score each response immediately instead of waiting for all N stim_q.get() → speculative cache
Pre-generate stimuli during response phase, validate after scoring flame.step(1) → flame.run_continuous()
Perpetual population loop with async score injection
