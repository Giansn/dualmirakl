PART 2: PIPELINE ARCHITECTURE (3 workers + fire-and-forget observer)
=====================================================================

  Three async workers connected by bounded queues:

    +--------------------+   stim_q (maxsize=2)   +--------------------+
    |  STIMULUS WORKER   |----------------------->|   SWARM WORKER     |
    |  Phase A           |                        |   Phase B          |
    |  (GPU 0 or 1)      |                        |   (GPU 0 + GPU 1)  |
    +--------------------+                        +---------+----------+
           ^                                                |
           |  (runs ahead: T+1 stimuli                      | resp_q (maxsize=2)
           |   generated during T's                         |
           |   Phase C on CPU)                              v
           |                                      +--------------------+
           |                                      |   SCORE WORKER     |
           |                                      |   Phase C (CPU)    |
           |                                      |   Phase F (GPU 2)  |
           |                                      +--------+-----------+
           |                                               |
           |                                               | on observer ticks
           |                                               | (every K ticks):
           |                                               v
           |                                      +--------------------+
           |                                      |   OBSERVER TASK    |
           |                                      |   Phase D (GPU 0)  |
           |                                      |   fire-and-forget  |
           |                                      +--------------------+
           |
           +--- [tick_done_q] -----> caller (sim_loop.py)


  Queue backpressure:
    stim_q maxsize=2 : stimulus worker can run 1-2 ticks ahead
    resp_q maxsize=2 : prevents memory buildup if scoring is slow
    tick_done_q      : unbounded, consumed by sim_loop.py main loop

PART 6: ADAPTIVE BALANCER DETAIL
==================================

  The balancer runs BETWEEN ticks (after score worker, before/during next stimulus).
  It uses GPU utilization (not power draw in v4) to decide rebalancing.

  +---------------------------------------------------+
  | AdaptiveBalancer                                   |
  |                                                    |
  | Every tick:                                        |
  |   1. GPUMonitor.sample()                           |
  |      - pynvml reads: util%, power, mem, temp       |
  |      - EMA smoothing (alpha=0.3)                   |
  |                                                    |
  |   2. Compute imbalance:                            |
  |      imbalance = (gpu0_ema_util - gpu1_ema_util)   |
  |                  / 100.0                            |
  |      Range: [-1.0, +1.0]                           |
  |      Positive = GPU 0 busier                       |
  |      Negative = GPU 1 busier                       |
  |                                                    |
  |   3. Deadband check:                               |
  |      if |imbalance| < 0.10 → no action             |
  |                                                    |
  |   4. Shift ONE participant per tick:               |
  |      imbalance > 0 → move 1 from auth → swarm     |
  |      imbalance < 0 → move 1 from swarm → auth     |
  |                                                    |
  |   5. KV cache cap enforcement:                     |
  |      max_auth = max_seqs - reserved                |
  |      reserved = 2 (observers)                      |
  |                + 1 (if env on authority)            |
  |      Cannot exceed this even if imbalance persists |
  |                                                    |
  |   6. Environment migration (after 3+ ticks):       |
  |      If authority consistently hotter → env stays  |
  |        on swarm (default)                          |
  |      If swarm consistently hotter → env moves to   |
  |        authority                                   |
  +---------------------------------------------------+

  Example: 6 participants, max_seqs=16

    Initial:  GPU 0: 3 participants + 2 observers = 5 seqs
              GPU 1: 3 participants + 1 environment = 4 seqs
              max_auth_participants = 16 - 2 = 14

    After T2:  GPU 0 at 78% util, GPU 1 at 55% util
               imbalance = +0.23 (> 0.10 deadband)
               Action: shift 1 participant auth → swarm
               New split: GPU 0: 2 parts, GPU 1: 4 parts

    After T3:  GPU 0 at 52% util, GPU 1 at 61% util
               imbalance = -0.09 (< 0.10 deadband)
               Action: none (within deadband)

    Convergence: typically 2-4 ticks to find stable split

PART 8: WHERE IDLE GAPS REMAIN
================================

  The pipeline eliminates most idle time, but some gaps persist:

  1. GPU 0 idle during Phase C (non-observer ticks)
     ------------------------------------------------
     Phase C runs entirely on CPU (e5-small-v2 embedding + score math).
     GPU 0 is idle unless the stimulus worker is generating T+1 stimuli.
     MITIGATION: stim_q backpressure lets stimulus worker run 1-2 ahead,
     filling this gap. But if A is faster than C, GPU 0 waits on put().

  2. GPU 1 during observer Phase D
     ------------------------------------------------
     Observer runs on GPU 0 only. GPU 1 is FREE.
     MITIGATION: this is the KEY v4 improvement. The pipeline immediately
     starts T+1's stimulus/swarm work. GPU 1 never waits for the observer.
     Only GPU 0 has partial overlap (observer vs next tick's participant batch).

  3. GPU 2 (FLAME) between sub-step bursts
     ------------------------------------------------
     FLAME runs 10 sub-steps per tick, then waits for the next tick's scores.
     FLAME step time << LLM inference time, so GPU 2 is idle most of the tick.
     MITIGATION: acceptable — FLAME is a population amplifier, not the bottleneck.
     Could increase sub_steps for finer resolution, but diminishing returns.

  4. CPU during Phase A and B
     ------------------------------------------------
     CPU does no heavy work during stimulus generation or participant responses.
     MITIGATION: none needed — CPU is the cheap resource. Gateway serves
     UI/API requests during this time.

  5. Startup cold-start (first tick only)
     ------------------------------------------------
     Tick 1 cannot overlap with "Tick 0" — pipeline is empty.
     All hardware is idle until the stimulus worker produces the first stimuli.
     MITIGATION: none. Amortized over the run; negligible for 20+ tick runs.


PART 9: FLAME GPU 2 DETAIL (per-tick cycle)
=============================================

  Within each dualmirakl tick, FLAME runs 10 sub-steps on GPU 2.
  Each sub-step has 3 CUDA layers:

    Sub-step 1 of 10:
    +--------------------------------------------------+
    | Layer 1: BROADCAST                                |
    |   Influencer agents output: (x, y, score, idx)    |
    |   Population agents output: (x, y, score, -1)     |
    |   Both write to shared Spatial2D message list      |
    |   (concurrent — different agent populations)       |
    +--------------------------------------------------+
              |
              v
    +--------------------------------------------------+
    | Layer 2: UPDATE                                   |
    |   Population agents read nearby messages (radius) |
    |   Weighted average: influencer_weight * inf_score  |
    |                   + 1.0 * pop_score               |
    |   Score dynamics:                                  |
    |     effective = score + susceptibility*(mean-score)|
    |     delta = alpha * (effective - score)            |
    |     new = score + delta*dampening*(1-resilience)   |
    |         + kappa*(peer_mean - score)                |
    |         + drift_sigma * noise                      |
    |     clamp to [0, 1]                                |
    +--------------------------------------------------+
              |
              v
    +--------------------------------------------------+
    | Layer 3: MOVE                                     |
    |   Population agents drift toward influencers       |
    |   + random walk (hash-based pseudo-random)         |
    |   Wrap at boundaries (toroidal space)              |
    +--------------------------------------------------+
              |
              v
    +--------------------------------------------------+
    | StepFunction: increment step_count                 |
    +--------------------------------------------------+

    ... repeat 9 more times ...

  Data bridge (score_worker in sim_loop):
    BEFORE sub-steps:
      bridge.push_influencer_scores(engine, [p.behavioral_score for p in participants])
    AFTER sub-steps:
      snapshot = bridge.pull_population_stats(engine, tick, sub_steps=10)
      → PopulationSnapshot(mean, std, min, max, histogram, influencer_scores)

  Observer integration:
    bridge.get_population_coupling_feedback(engine)
      → {mean_score, std_score, polarization, center_offset, n_population}
      Injected into observer_a's world state as flame_feedback


PART 10: COMPARISON — v3 (sequential) vs v4 (harmony pipeline)
================================================================

  v3 (run_tick — sequential within each tick):
  +------+------+------+------+------+------+------+------+
  |  A1  |  B1  |  C1  |  A2  |  B2  |  C2  |  A3  |  B3 ...
  +------+------+------+------+------+------+------+------+
  GPU 0:  A1     B1(3p)  idle   A2     B2(3p)  idle   ...
  GPU 1:  idle   B1(3p)  idle   idle   B2(3p)  idle   ...
  CPU:    idle   idle    C1     idle   idle    C2     ...
  GPU 2:  idle   idle    F1     idle   idle    F2     ...

  Utilization: ~40% per GPU (large idle gaps during C and between ticks)


  v4 (GPU Harmony — pipelined, overlapped):
  +------+------+------+------+------+------+------+------+------+------+
  |  A1  |  B1  | A2+C1| B2+F1| A3+C2| B3+F2|A4+C3|B4+F3| D4+C4| B5...
  +------+------+------+------+------+------+------+------+------+------+
  GPU 0:  A1     B1(3p)  A2     B2(3p)  A3     B3(3p)  A4     B4(3p) D4  A5  B5...
  GPU 1:  idle   B1(3p)  idle   B2(3p)  idle   B3(3p)  idle   B4(3p)     B5(3p)...
  CPU:    idle   idle    C1     idle    C2     idle    C3     idle   C4   idle...
  GPU 2:  idle   idle    idle   F1     idle    F2     idle    F3    F4    idle...

  Utilization: ~65-75% per GPU (Phase A fills Phase C gaps, observer is async)


  Throughput gain: ~1.6-1.8x ticks/second (measured, depends on model + hardware)
  The gain comes from:
    1. Stimulus worker fills GPU 0 during CPU-only Phase C
    2. Observer is fire-and-forget — GPU 1 never blocks
    3. FLAME runs in executor thread — CPU stays free for C+1
